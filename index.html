Thread Dump Doctor — Free Java Thread Dump Analyzer

Thread Dump Doctor is a free, open-source, privacy-first tool for analyzing Java thread dumps directly in your browser. No data is ever uploaded or sent to a server. It works fully offline after the page loads.

What is a Java thread dump?

A Java thread dump is a snapshot of all threads running in a JVM at a given moment. It includes each thread's name, state (RUNNABLE, WAITING, TIMED_WAITING, BLOCKED), and full stack trace. Thread dumps are produced using jstack, kill -3, or VisualVM and are the primary diagnostic tool for investigating Java application hangs, deadlocks, and high CPU usage.

What problems does Thread Dump Doctor detect?

  • Java deadlocks — JVM-reported deadlocks and classic A→B→A lock-cycle deadlocks where two or more threads are each waiting for a lock held by the other.
  • CompletableFuture self-deadlock (pool self-starvation) — threads inside a thread pool submitting tasks back to the same pool and blocking with .get() or .join(), exhausting all workers (Triangle B pattern).
  • Socket-read I/O starvation — pool threads blocked on blocking socket reads (SocketInputStream.read, NioSocketImpl.read), preventing other work from proceeding (Triangle C pattern).
  • Callback-thread hijacking — user callbacks running on upstream I/O threads (OkHttp, Netty), delaying I/O processing for the entire application.
  • In-callback blocking — .join() or .get() calls inside CompletableFuture callbacks, which can deadlock the entire callback chain.
  • Thread pool saturation — all workers in a named thread pool are active with zero idle threads, preventing new work from being accepted.
  • ForkJoinPool.commonPool() overload — the JVM's shared pool is heavily occupied, starving unrelated parallelStream() calls and other JDK consumers.
  • Unbounded WAITING threads — application threads waiting indefinitely with no timeout, which can cause resource leaks and invisible hangs.
  • Four-bucket triage — every thread is classified into one of four categories: noise (GC, VM threads), framework-idle (waiting in pool queues), app-working (RUNNABLE user code), or app-stuck (BLOCKED/WAITING in user code).

How to use Thread Dump Doctor

  1. Generate a thread dump using jstack <pid> or kill -3 <pid> and save it as a .txt file.
  2. Visit Thread Dump Doctor at https://kushagrasri.github.io/thread-dump-doctor
  3. Drag and drop the file onto the upload area, or click to browse.
  4. Click the Analyze button. Results appear within milliseconds.
  5. Review severity-ranked findings (Critical, Warning, Info) with recommended actions.

Two-dump diff mode

Upload two thread dumps taken approximately 30 seconds apart. Thread Dump Doctor compares them and highlights threads that have not made any stack progress between the two snapshots. This is the most reliable way to confirm a live stall or deadlock, as opposed to a momentary pause.

Frequently asked questions

Is Thread Dump Doctor free?

Yes. Thread Dump Doctor is completely free. There is no sign-up, no account, and no cost. It is open source.

Is my thread dump data private?

Yes. All analysis runs locally in JavaScript inside your browser. Your thread dump file is never sent to any server or third party. The tool works fully offline after the initial page load.

What thread dump formats are supported?

Thread Dump Doctor supports OpenJDK jstack output, Oracle JDK jstack output, HotSpot kill -3 signal output, and VisualVM thread dump exports. Partial support for GraalVM native image thread dumps.

How is Thread Dump Doctor different from other thread dump analyzers?

Thread Dump Doctor runs 100% in the browser with no server upload required, making it safe for analyzing production thread dumps containing sensitive class names and application data. It specifically detects complex patterns like CompletableFuture self-deadlock and callback-thread hijacking that most other tools miss.

What is pool self-starvation?

Pool self-starvation occurs when all threads in a thread pool are blocked waiting for results of tasks that are queued behind them in the same pool. Because all workers are occupied waiting, the queued tasks can never run, creating a deadlock. This is especially common with CompletableFuture chains that use a fixed-size executor.

How do I generate a thread dump?

On Linux/macOS: run jstack <pid> > dump.txt or kill -3 <pid> (output goes to the JVM's stdout/stderr). On Windows: use jstack <pid> or VisualVM. In containers: use jcmd <pid> Thread.print.

Technology
